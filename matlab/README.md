# MATLAB / Simulink — DC Power Supply App

The MATLAB/Simulink side of the Agnipariksha DC power-supply app: a plant
model, a Simulink live console, an App Designer GUI, and OPC UA / REST
bridges to the backend. **`RUN_ME.m` is the single entry point.**

MATLAB/Simulink can't run in this repo's Linux CI, so the **numerical model
is executed headlessly in GNU Octave** and pinned to a shared reference trace
(`reference_trace.csv`) generated by the Python model — the App Designer GUI
and Simulink wiring still need genuine MATLAB.

## Entry point

```matlab
cd matlab; RUN_ME      % menu: App Designer console / Simulink console / plant
```

## Simulink interface

`simulink/build_psu_console_model.m` assembles a Simulink model that is the
operator console: setpoint sources → a **PSU Live Interface** MATLAB Function
block (`psu_live_interface.m`, which calls the backend over REST) → Scope +
Displays for V/I/P/Tj. Swap the setpoint Constants for Dashboard **Knob** /
**Rocker Switch** blocks and drop Dashboard **Gauge** blocks on the outputs
for a full interactive HMI driven entirely from Simulink. The plant-only
simulation model is `build_psu_plant_model.m`.

## The model

A discrete first-order DC-PSU plant, identical to
`backend/app/opcua_bridge.py::DemoPsuSource`:

```
v[k]  = v[k-1]  + response * (Vsp·OutEn - v[k-1])
i[k]  = i[k-1]  + response * (Isp·OutEn - i[k-1])
P[k]  = v[k] · i[k]
Tj[k] = Tj[k-1] + thermal_gain·P[k] - cooling·(Tj[k-1] - ambient)
```

Defaults: `response = 0.35`, `thermal_gain = 0.02 °C/W`, `cooling = 0.05`,
`ambient = 25 °C`. Internal state is full precision; reported readings are
rounded (V/I/P to 4 dp, Tj to 3 dp) to match the Python observable output.

## Keeping MATLAB and Python in lock-step

`reference_trace.csv` is regenerated on the Python side with:

```bash
python -m backend.app.psu_trace
```

and guarded in CI by `backend/tests/test_psu_trace.py` (fails if the Python
model drifts from the committed CSV). On the MATLAB side,
`run_psu_step_response.m` loads the same CSV and asserts the MATLAB plant
matches it to within `1e-3` (a small tolerance absorbs MATLAB-vs-Python
round-half differences at the last decimal). If you change a plant constant,
change it in **both** `psu_plant_defaults.m` and `DemoPsuSource`, then
regenerate the CSV.

## Verified execution (no MATLAB license needed)

The numerical plant model is **executed in CI-style headlessly under GNU
Octave** (a free MATLAB-compatible interpreter) and checked against the
Python reference — so the MATLAB code is actually run, not just eyeballed:

```bash
# Plant step model:
octave-cli -q --eval "cd matlab/tests; ok = plant_parity_check();   exit(double(~ok))"
# ticks=60  maxAbsDiff=0.000e+00  -> PASS

# Simulink block-diagram design (recurrences + Unit Delay initial conditions):
octave-cli -q --eval "cd matlab/tests; ok = simulink_plant_parity(); exit(double(~ok))"
# simulink-design ticks=60  maxAbsDiff=0.000e+00  -> PASS
```

`tests/plant_parity_check.m` runs the real `psu_plant_step.m` for 60 ticks;
`tests/simulink_plant_parity.m` runs the exact block-by-block recurrence that
`build_psu_plant_model.m` wires in Simulink (including the Unit Delay initial
conditions). Both assert agreement with `reference_trace.csv` to within
`1e-3` (observed: exact, `0.0`), in MATLAB or Octave. What still needs genuine
MATLAB: the App Designer GUI (`uifigure`), the Simulink `.slx` wiring, and Web
App Server packaging — Octave has none of those toolboxes.

## Quick start

```matlab
% 1. Verify the plant matches the Python reference and see the step response
run_psu_step_response

% 2. Build the Simulink model (requires Simulink)
build_psu_plant_model            % writes psu_plant_model.slx

% 3. Drive the live OPC UA server from MATLAB (server must be running;
%    requires Industrial Communication Toolbox)
sp = struct('voltage_v',48,'current_a',2,'output_enabled',true);
o  = struct('username','operator','password','pv6000');   % omit for open server
d  = psu_opcua_bridge(sp, o)
```

## OPC UA integration

`psu_opcua_bridge.m` is a thin OPC UA **client** against the same server the
web/standalone app uses (`backend/app/opcua_server.py`). It writes the three
client-writable Setpoint nodes and reads the Readings/Info nodes, so a MATLAB
controller can close the loop over the standard OPC UA address space. Auth is
optional and matches the server's `users=` profile (Step 2).
